3. Memory System Architecture (MemPalace Integration)
Location: scripts/memory/ + knowledge_graph/mempalace/
Components:
1. DatabaseSchema (memory/schema.py):
   - SQLite tables: actions, patterns, knowledge, insights, metadata
   - Schema version 2
   - Indexes on timestamp, type, entity, etc.
   - WAL mode enabled
2. MemoryStore (memory/__init__.py):
   - Coordinates episodic, semantic, procedural memory
   - Single SQLite database at scripts/memory/memory.db
   - Provides learn(), recall(), suggest() APIs
3. EpisodicMemory (memory/episodic.py):
   - Stores action-outcome episodes
   - "What happened when we did X"
   - Used by planner for learning from experience
4. SemanticMemory (memory/semantic.py):
   - Entity-relationship storage
   - Integrates with KG for semantic queries
   - Stores facts with confidence scores
5. ProceduralMemory (memory/procedural.py):
   - Learned patterns: issue → fix
   - Success rates, confidence scoring
   - Pattern definitions from utils/patterns.py
Data Flow:
Planner Agent → MemoryStore.learn(action, outcome)
              → episodic.store(episode)
              → semantic.add_knowledge()
              → procedural.learn_fix()
Planner Agent → MemoryStore.recall(query)
              → semantic.search()
              → procedural.suggest()
              → episodic.get_similar()
Status: The memory system is well-architected but likely underutilized. Check scripts/memory/actions.json and patterns.json for actual data.
Observation: scripts/memory/actions.json was modified recently (M status). Check its contents to see if learning is happening.

4. Entity Resolver: Fuzzy Name Matching with Ollama
File: scripts/resolve/entity_resolver.py
Purpose: Resolve scraped MP names to canonical entities in the vault.
Approach:
1. Build canonical database from vault files (by reading all politician markdown)
2. Try exact match (after normalizing diacritics, case)
3. Try fuzzy matching (Levenshtein)
4. If confidence low, use Ollama (phi3 or qwen2.5-coder) to disambiguate
5. Returns MatchResult with canonical_id and confidence

13. Concurrent Write Safety
Pattern: Atomic writes (atomic_write, atomic_json_write) + file locking not used.
Risk: Two scrapers writing same session file simultaneously could still conflict if they both succeed in temp.replace(). But replace() is atomic on POSIX, so whichever writes last wins. No corruption, but lost updates possible.
Better: Use file locking (fcntl.flock) or write to separate temp dirs then atomic rename with unique names.
entities.json: Multiple writers possible (merge_vault_to_kg.py, stenomd_master.py, agents?). Need to check if they all use atomic_json_write.

14. Caching Strategy
Request Cache (scrapers):
- In-memory dict REQUEST_CACHE with (url, params): (response, expires_at)
- TTL 3600 seconds
- Lives only for the process (not shared)
- Good for single run, but doesn't persist across runs
Session Cache (DataValidator):
- Loads all existing sessions at init
- _existing_sessions dict
- refresh_sessions() to reload
- Speeds up duplicate detection significantly
Recommendation: Consider persistent request cache (Redis/SQLite) to avoid re-fetching same pages across runs. But respect robots.txt and rate limits.
